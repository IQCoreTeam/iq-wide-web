# iq-wide-web

월드 와이드 웹을 솔라나 네이티브로 다시 풀어낸 것.

WWW가 HTTP 서버들의 네트워크를 통해 정보를 엮었다면, iq-wide-web은 솔라나 체인의 트랜잭션을 엮는다. 같은 충동(리소스를 식별자로 가리키고, 사람이 기억할 만한 이름을 붙이고, 리소스끼리 링크하고, 발견하고 검색하고, 누구나 호스팅할 수 있게 하기)을 다른 매체에서 다시 구현한 것.

사람을 위한 진입점: 어떤 식별자든 던지면 — 지갑이든 PDA든 트랜잭션 시그니처든 `.sol` 도메인이든 — 그게 뭔지 알아내서 맞는 화면을 보여준다.

WWW와의 차이는 데이터의 영속성. 콘텐츠가 체인 위에 살기 때문에 호스팅(=게이트웨이)이 사라져도 콘텐츠는 남는다. 게이트웨이는 그저 캐시일 뿐 — 누구든 새로 띄울 수 있다.

> **현재 상태: 기획 중.** 아직 코드는 없음. 아래는 만들고 있는 것에 대한 설명이다.

## 정체

`gateway.iqlabs.dev`는 지금 한 자리에서 두 가지를 동시에 한다:

1. JSON / raw bytes 응답 — 기계용 (SDK, 봇, dApp)
2. HTML / 이미지 / 사이트 서빙 — 사람용 (브라우저 방문)

`iq-wide-web`은 사람을 향한 절반을 떼어낸 별도의 서비스다. 게이트웨이는 데이터 API이고, iq-wide-web은 그 데이터를 받아 페이지로 그려준다. 다르게 말하면 iq-wide-web은 *어떤* 게이트웨이 호출을 할지 결정하고 결과를 한 화면으로 엮어내는 레이어.

기존 `iqprofile.net`은 이 서비스의 한 뷰(지갑 프로필)로 흡수된다.

## 무엇을 해결하나

지금은 누가 SNS 도메인(`zo.sol`)을 방문하면 게이트웨이가 302로 던지고 주소창이 `gateway.iqlabs.dev/...`로 바뀐다. 그 `.sol`을 산 사람 입장에서는 자기 도메인 정체성이 막 증발해버린 셈.

앞에 iq-wide-web을 두면 콘텐츠가 같은 호스트 아래에서 서빙된다 — 주소창은 `browser.iqlabs.dev/zo.sol`에 그대로 머무른다.

또 다른 한 가지: 식별자 디스패치. 식별자 하나만 받아도 iq-wide-web이 그게 뭔지 분류해서 맞는 걸 보여준다. 사용자는 어떤 PDA가 git repo인지 세션인지 다른 무엇인지 알 필요 없다.

### 구체적인 예 — SNS 도메인 소유자의 두 가지 시나리오

누군가 `zo.sol`을 갖고 있고 페이지를 세팅하고 싶다고 치자. 의도가 두 갈래로 갈린다. 게이트웨이만 있는 세계와 iq-wide-web이 있는 세계를 시나리오별로 비교.

**시나리오 A — 개인 프로필 페이지 (지갑)**

지갑을 정체성으로 표현하는 페이지: UserInventory 메타데이터(이름 / 자기소개 / 사진), 최근 자산, 게시물.

게이트웨이만 있을 때:
- SNS 레코드가 지갑을 가리키는 패턴이 정의돼 있지 않다 (`/site/{sig}/{file}`은 사이트용 패턴)
- 결국 `iqprofile.net/{wallet}` 같은 걸 외부에서 손수 링크함
- "프로필 페이지"가 SNS와 일급으로 묶이지 못함

iq-wide-web이 있을 때:
- SNS 레코드에 지갑 주소만 넣어둠
- 방문자가 `zo.sol`을 치면 → iq-wide-web이 지갑임을 인식 → 프로필을 렌더
- 주소창은 `zo.sol.site`에 그대로

**시나리오 B — git 기반 사이트 (커밋이 곧 페이지)**

소유자의 최신 git 커밋을 소스로 삼고 iqpages에 등록된 사이트.

게이트웨이만 있을 때:
- SNS 레코드가 `https://gateway.iqlabs.dev/site/{sig}/{file}`을 들고 있다
- 그 레코드는 새 커밋이 들어와도 자동 갱신되지 않음 — **하나의 sig에 고정된 스냅샷**
- 새 커밋을 SNS에 반영하려면 소유자가 직접 새 sig로 레코드를 갈아 끼워야 함
- 처음부터 "최신 커밋 자동 추적"을 원해도 그런 메커니즘이 없음
- 버전 감각이 어색함 — SNS가 얼어붙은 한 커밋에 묶여있음

iq-wide-web이 있을 때:
- SNS 레코드에 commit-table PDA(모든 커밋의 부모)를 넣어둔다
- 방문자가 `zo.sol`을 치면 → iq-wide-web이 PDA를 인식 → iqpages 등록 확인 → 최신 커밋의 엔트리를 서빙
- 새 푸시가 들어오면 다음 방문에서 즉시 반영 — SNS 레코드 수정 없음
- 주소창은 `zo.sol.site`에 그대로

두 시나리오의 공통된 가치는: **SNS 도메인이 안정된 정체성으로 동작한다는 것**. 뒤에 있는 콘텐츠는 바뀔 수 있지만 도메인 자체는 그대로 있다.

### 왜 게이트웨이 안에 만들지 않나

원래 계획은 그냥 게이트웨이에 얹어 짓는 거였다. 멈춘 이유는 단순:

> **API 코드와 프론트엔드 페이지 코드를 한 서비스에 섞으면 뭐가 뭔지 분간이 안 된다.**

이건 취향 문제가 아니라 web2에서 충분히 검증된 분리. 평범한 스택을 들여다보면 컨테이너/서비스가 두 개로 나뉜다:

- **백엔드 컨테이너** — `api.example.com` 같은 거. Express / FastAPI / Spring, JSON 응답. 기계들(모바일 앱, 다른 서비스, JS 클라이언트)이 호출.
- **프론트엔드 컨테이너** — `www.example.com` 또는 `app.example.com`. Next.js / Nuxt / Vite, HTML / CSS / JS / 이미지를 서빙. 사람이 브라우저로 방문.

왜 분리하나? 두 코드베이스의 **변경 주기, 빌드 도구, 스케일링 패턴, 캐싱 정책, 의존성 그래프, 배포 빈도**가 다 다르기 때문이다. 한 데 묶으면 API 한 줄 고치겠다고 프론트 빌드 다시 돌리거나, 프론트가 색깔 바꿨다고 API 컨테이너 재배포하게 된다. 대부분의 팀이 오래전에 답을 냈다 — *"백엔드 서비스 + 프론트엔드 앱"* 이 표준.

여기도 같은 길. 게이트웨이는 백엔드 데이터 서비스, iq-wide-web은 프론트엔드 웹 앱. 같이 살 때 생기는 마찰을 구체적으로:

- 파일 트리가 매번 "이게 API 핸들러야 화면 컴포넌트야?" 퍼즐이 됨
- 라우팅이 충돌. 게이트웨이의 URL은 API 경로(`/table/...`, `/data/...`)인데 사람용 URL은 페이지 경로(`/{wallet}`, `/{repo}.sol`)다. 같은 호스트 위에서 요청마다 "이건 API야 페이지야?"를 결정해야 함. 패턴이 겹치는 순간 HTML 페이지가 기계 응답으로 새거나 그 반대가 벌어짐

게다가 iq-wide-web이 정말로 되고 싶은 건 *"식별자 분류해서 보여주기"* 가 아니다 — **dApp 데이터를 엮어서 탐험할 수 있게 만드는 웹**이다 (마인드맵, 검색, 프로필↔repo↔트랜잭션 링크). 그건 읽기 전용 HTTP 캐시와는 다른 모양.

그래서 게이트웨이는 데이터 서비스로 남고, iq-wide-web이 그 위의 사람용 웹.

## 식별자 종류

`browser.iqlabs.dev/{ident}` — `{ident}`는 다음 중 하나:

- **`.sol` 도메인** — SNS로 풀어낸 뒤, 그 결과로 다시 디스패치
- **pubkey (32–44자 base58)** — 지갑 또는 PDA
- **tx 시그니처 (87–88자 base58)** — 트랜잭션 한 건

첫 디스패치는 모양 기반(RPC 없음). 필요하면 게이트웨이 한 번 호출로 PDA 종류(지갑 / Table / UserInventory / Session / …)를 풀고, 각 다운스트림 서비스의 자기 repo를 보고 맞는 뷰를 고름.

## 호스트

| 호스트 | 역할 |
|---|---|
| `browser.iqlabs.dev` | 메인 진입점 |
| `profile.iqlabs.dev` | 레거시 지원 — 기존 `iqprofile.net` URL이 그대로 동작. 같은 서비스, 같은 화면 |

별개의 앱이 아님. 한 배포가 두 호스트 모두에 응답.

## 무엇이 렌더되나

식별자 종류별로:

- **지갑** — 프로필 페이지 (이름, 자기소개, 최근 자산 / 게시물)
- **iqpages에 등록된 git repo PDA** — 최신 커밋의 엔트리 페이지(정적 사이트)
- **iqpages에 등록 안 된 git repo PDA** — 커밋 히스토리, 파일 트리, "iqpages에 등록하기" 프롬프트
- **Session PDA** — 번들 청크 뷰어 + 다운로드
- **트랜잭션 시그니처** — 자산 미리보기(이미지 / 텍스트 / 파일) + 다운로드 + 메타데이터
- **그 외** — JSON 뷰 또는 404

## 데이터 소스

모든 온체인 데이터는 `gateway.iqlabs.dev`를 통한다. 이 서비스에서 직접 RPC 호출하지 않음 — 게이트웨이가 이미 캐싱과 점진적 갱신을 다 한다.

## 새 아이디어 — 될 수도 안 될 수도 있음. 문서 자체가 초안이지만, 이 부분은 더 초안.

### 1. 마인드맵 (Nubs의 아이디어)

Nubs가 `gateway.iqlabs.dev`에 모든 DbRoot를 한 번에 돌려주는 엔드포인트를 추가하고 있다. 각 DbRoot가 자기 dApp의 테이블 시드(=hint)를 들고 있고, hint 구조 자체가 dApp별로 의미를 담음(iqchan: board → thread → reply, git: repo → commits, iqpages: deployed).

모든 hint 트리를 가져오면 **IQ 위에 있는 모든 dApp의 전체 구조**가 한눈에 들어온다. iq-wide-web은 그걸 Obsidian처럼 시각화할 수 있음 — 노드는 DbRoot / Table / Row, 간선은 hint에 인코딩된 부모-자식 관계 (예: 어떤 리플라이가 어떤 스레드에 속하는지).

웹의 가치가 *"하이퍼링크가 페이지들을 묶는다"* 였다면, IQ의 가치는 *"온체인 데이터가 hint 구조를 통해 스스로 묶인다"*. 그 짜임을 보이게 만들면 정보 네트워크 자체가 탐색 가능한 것이 된다.

### 2. 키워드 검색 (구글 스타일)

마인드맵이 *"전체 구조 둘러보기"* 라면, 검색은 *"키워드로 바로 뛰어들기"*. 키워드를 치면 IQ 전체에서 매칭되는 인스크립션들의 랭킹된 리스트가 뜸 — 구글 검색 결과 페이지 같은 느낌.
